# Agent Tool Reference

Tài liệu này mô tả toàn bộ công cụ mà Agent có quyền sử dụng hoặc cần hiểu kết quả của chúng.

---

## Pipeline Overview

```
[HUMAN] recon.sh + endpoint-filter.py
    → target_info/site-endpoints.txt

[HUMAN] form-auto-submit.py
    → target_info/site-forms.txt  (context cho Agent khi tạo payload)

[AGENT] use-ZAP.py  +  idor_exploit.py  (chạy song song)
    → zap_results.json  (DO NOT READ — quá nặng)
    → idor_results.json (AGENT đọc file này nếu có)

[AGENT] filter-finding.py
    → zap_filtered.json

[AGENT] zap-raw-res-reducer.py
    → zap_reduced.json  ← Agent đọc file này

[AGENT] Đọc target_info/, đọc zap_reduced.json, phân tích
    → Tạo manifest + gọi agent-scaner.py bulk-send
    → Gọi agent-scaner.py search
    → Viết report
```

## Files Agent Được Đọc

### `target_info/target-address.txt`

Base URL của target. Dùng để ghép full URL khi tạo manifest.

```
http://192.168.153.200/DVWA/
```

### `target_info/site-endpoints.txt`

Danh sách endpoint đã lọc, 1 path mỗi dòng (path tương đối hoặc full URL).

```
/vulnerabilities/xss_s/
/vulnerabilities/sqli/
/vulnerabilities/fi/?page=include.php
```

### `target_info/site-forms.txt`

Form HTML thô + URL submit tương ứng. Dùng để biết chính xác tên param và method khi tạo payload.

```
<form method="post" name="guestform">
  <input name="txtName" type="text"/>
  <textarea name="mtxMessage"></textarea>
  <input name="btnSign" type="submit" value="Sign Guestbook"/>
</form>
_
http://192.168.153.200/DVWA/vulnerabilities/xss_s/
```

### `target_info/session.txt`

Session hiện tại. Truyền vào `-s` cho các tool.

**Quy ước vị trí — 2 dòng:**

```
PHPSESSID=abc123; security=low   ← dòng 1: ATTACKER (bắt buộc — dùng cho toàn bộ workflow)
PHPSESSID=xyz789; security=low   ← dòng 2: VICTIM   (tùy chọn — bỏ trống → skip IDOR check)
```

- Dòng 1 được dùng bởi: `use-ZAP.py`, `agent-scaner.py`, `idor_exploit.py` (attacker role)
- Dòng 2 chỉ dùng bởi: `idor_exploit.py` (victim role — tài khoản bị thử truy cập trái phép)

## `zap_reduced.json` — ZAP Results (READ ONLY)

Output của `zap-raw-res-reducer.py`. Agent đọc file này để xác định findings và identify endpoints cần custom probe.

### Schema

```json
{
	"base": "http://192.168.153.200/DVWA/",
	"items": [
		{
			"target": {
				"endpoint": "vulnerabilities/xss_s/",
				"method": "POST",
				"inputVector": "form",
				"param": "txtName"
			},
			"findings": ["Cross Site Scripting (Persistent) [High/Medium]"],
			"payloads": ["</div><script>alert(1)</script><div>"],
			"response": "HTTP/1.1 200 OK\nContent-Type: text/html;charset=utf-8\n\n[ZAP Evidence] <script>alert(1);</script>\n[Context] ...Hello <script>alert(1);</script> World...\n[Reflection] ...name=txtName value=\"</div><script>...",
			"repeat": 5
		}
	]
}
```

### Field Notes

| Field                | Ý nghĩa                                                                                                                     |
| -------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| `target.endpoint`    | Path tương đối so với `base`. Ghép `base + endpoint` để có full URL.                                                        |
| `target.inputVector` | Nơi inject: `form`, `url`, `header`, v.v.                                                                                   |
| `target.param`       | Tên parameter bị inject.                                                                                                    |
| `findings`           | `"<Type> [Severity/Confidence]"` — có thể nhiều finding trên 1 target.                                                      |
| `payloads`           | Payload ZAP đã thử (tương ứng 1-1 với `findings`). **Không thử lại các payload này.**                                       |
| `response`           | Response đã rút gọn: status line + headers quan trọng + `[ZAP Evidence]` + `[Context]` + `[Reflection]`. Không có raw body. |
| `repeat`             | Số lần ZAP báo finding trên cùng cluster. Càng cao → confidence càng cao. Chỉ xuất hiện khi ≥ 2.                            |

### Agent Decision Logic

- **Confirmed (repeat ≥ 2, severity High/Medium)** → Ghi finding, tạo 1-2 payload bổ sung để lấy thêm evidence nếu cần.
- **Low confidence (Confidence = Low/Medium, repeat = 1)** → Ưu tiên probe thêm.
- **Không có finding** nhưng endpoint quan trọng (có param, status ≠ 200) → Tự suy và probe.
- **Status 403/WAF suspect** → Thử bypass header payload.

## `agent-scaner.py` — Targeted Probe Tool

**Công cụ cho Agent tự gọi.** Gửi custom request và tìm keyword trong response.

### STRICT RULES

- **MAX 3 lần gọi `bulk-send`** trong toàn bộ session.
- **MAX 3 payload per endpoint** mỗi lần gọi.
- **KHÔNG đọc file response thô** (`tmp_responses/*.txt`). Chỉ đọc qua `search`.
- Mỗi lần `bulk-send` phải theo ngay bởi `search` trước khi gọi `bulk-send` tiếp theo.

---

### COMMAND: `bulk-send`

Gửi hàng loạt request theo manifest JSON, lưu response body xuống disk.

```bash
py tools/agent-scaner.py bulk-send -m <manifest.json> -s <session_file>
```

**Input — `manifest.json`:**

```json
[
	{
		"url": "http://192.168.153.200/DVWA/vulnerabilities/sqli/",
		"method": "GET",
		"param": "id",
		"values": ["1' OR '1'='1", "1' AND SLEEP(0)--", "1 UNION SELECT null--"]
	},
	{
		"url": "http://192.168.153.200/DVWA/vulnerabilities/xss_r/",
		"method": "GET",
		"param": "name",
		"values": ["<script>alert(1)</script>", "<img src=x onerror=alert(1)>"]
	}
]
```

| Field    | Mô tả                                    |
| -------- | ---------------------------------------- |
| `url`    | Full URL (base + endpoint).              |
| `method` | `"GET"` hoặc `"POST"`.                   |
| `param`  | Tên parameter inject.                    |
| `values` | Danh sách payload. **Tối đa 3 phần tử.** |

**Output — stdout (JSON array):**

```json
[
	{
		"endpoint": "http://192.168.153.200/DVWA/vulnerabilities/sqli/",
		"method": "GET",
		"param": "id",
		"value": "1' OR '1'='1",
		"response_id": "a1b2c3d4",
		"status_code": 200,
		"ms": 45,
		"len": 3821
	},
	{
		"url": "http://...",
		"value": "...",
		"error": "Connection timeout"
	}
]
```

| Field         | Ý nghĩa                                                          |
| ------------- | ---------------------------------------------------------------- |
| `response_id` | ID 8 ký tự hex để tra cứu bằng `search`.                         |
| `status_code` | HTTP status. Bất thường (403, 500) = signal đáng chú ý.          |
| `ms`          | Response time (ms). Tăng bất thường → nghi time-based injection. |
| `len`         | Response body length. Thay đổi lớn giữa các payload → signal.    |
| `error`       | Xuất hiện thay cho các field trên nếu request thất bại.          |

---

### COMMAND: `search`

Tìm keyword trong các response đã lưu. **Auto-delete file sau khi tìm.**

```bash
py tools/agent-scaner.py search --ids <id1> [id2 ...] -k <kw1> [kw2 ...]
```

| Arg               | Mô tả                                                     |
| ----------------- | --------------------------------------------------------- |
| `--ids`           | Một hoặc nhiều `response_id` từ `bulk-send`.              |
| `-k / --keywords` | Keyword cần tìm (case-insensitive). Có thể nhiều keyword. |
| `--no-cleanup`    | Giữ file response sau search (debug). Mặc định: tự xóa.   |

**Output — stdout (JSON object):**

```json
{
	"a1b2c3d4": {
		"hits": [{ "kw": "sql syntax", "snip": "...error in your SQL syntax near '1' OR '1'='1' at line 1..." }]
	},
	"e5f6a7b8": {
		"hits": []
	}
}
```

| Field         | Ý nghĩa                                                                        |
| ------------- | ------------------------------------------------------------------------------ |
| `hits`        | Danh sách match. Empty list = không tìm thấy keyword.                          |
| `hits[].kw`   | Keyword đã khớp.                                                               |
| `hits[].snip` | Đoạn text xung quanh vị trí match (~180 ký tự). Dùng để confirm vulnerability. |

**Keyword Strategy:**

| Vulnerability           | Keywords gợi ý                                                  |
| ----------------------- | --------------------------------------------------------------- |
| SQLi                    | `sql syntax`, `mysql_fetch`, `ORA-`, `SQLSTATE`, `syntax error` |
| Blind SQLi              | _(dùng `ms` diff thay vì keyword)_                              |
| XSS Reflected           | Chính xác payload đã gửi, hoặc tag `<script`, `onerror`         |
| Path Traversal / LFI    | `root:`, `[boot loader]`, `daemon:`, `bin/bash`                 |
| RCE / Command Injection | `uid=`, `www-data`, `root`, output lệnh cụ thể                  |
| SSTI                    | Kết quả phép tính (vd: gửi `{{7*7}}` → tìm `49`)                |

---

### Agent Workflow

```
1. Đọc target_info/target-address.txt → lấy base URL
2. Đọc target_info/site-forms.txt → biết param name chính xác
3. Đọc zap_reduced.json → phân loại findings

4. Lần 1 — Probe high-priority targets:
   → Tạo manifest.json (max 3 payload/endpoint)
   → bulk-send → lấy response_ids + metadata
   → search với keyword phù hợp → đọc snippets
   → Ghi nhận confirmed / cần probe thêm

5. Lần 2 (nếu cần) — Probe targets low-confidence:
   → Tạo manifest mới (payload khác, không trùng ZAP + lần 1)
   → bulk-send → search

6. Lần 3 (nếu cần) — Edge cases / bypass attempt

7. Viết report:
   - Confirmed findings: endpoint + param + payload + evidence snippet + severity
   - Unconfirmed (cần manual verify): endpoint + lý do nghi ngờ
   - Hướng fix cho từng loại lỗ hổng
```

## `idor_exploit.py` — IDOR Verification Tool

**Chạy song song với Phase 1A (ZAP scan).** Kiểm tra các endpoint có tham số ID để phát hiện IDOR bằng cách so sánh response giữa 2 session.

```powershell
$env:PYTHONUTF8=1; py tools/idor_exploit.py `
  -e target_info/site-endpoints.txt `
  -b <base_url> `
  -s target_info/session.txt `
  -o idor_results.json
```

| Arg                     | Mô tả                                               |
| ----------------------- | --------------------------------------------------- |
| `-e` / `--endpoints`    | File endpoint paths (tương đối). Yêu cầu kèm `-b`.  |
| `-b` / `--base-url`     | Base URL của target.                                |
| `-u` / `--urls`         | File full URLs (thay thế cho `-e -b`).              |
| `-s` / `--session`      | `session.txt` — dòng 1 = attacker, dòng 2 = victim. |
| `-o` / `--output`       | Output JSON (default: `idor_output.json`).          |
| `--span`                | Khoảng ID thử xung quanh giá trị gốc (default: 2).  |
| `--min-id` / `--max-id` | Giới hạn ID (default: 1–5).                         |

**Logic phát hiện IDOR:**

- Trích xuất ID params từ URL (id, user_id, uid, username, ...)
- Với mỗi ID: gửi request bằng attacker session
- Nếu response chứa identity của victim → **confirmed IDOR**
- Nếu response khác nhau giữa 2 session → **candidate IDOR**

**Output — `idor_results.json`:**

```json
{
	"schema_version": "1.0",
	"target": "http://...",
	"idor_findings_count": 1,
	"idor_findings": [
		{
			"compact_summary": {
				"endpoint": "/path/?param=",
				"status": "confirmed",
				"confidence": "high",
				"reason": "cross-session unauthorized object access"
			},
			"full_verification": {
				"authenticated_as": "attacker",
				"requested_id": 2,
				"object_owner": "victim_username",
				"access_result": "success",
				"clean_diff": "...",
				"authorization_bypass_type": "horizontal"
			}
		}
	]
}
```

**Nếu dòng 2 (victim) trống:** Tool xuất file rỗng (`idor_findings_count: 0`) và thoát sạch — pipeline không bị gãy.
